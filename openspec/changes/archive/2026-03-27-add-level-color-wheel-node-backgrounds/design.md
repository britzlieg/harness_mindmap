## Context

当前节点默认背景色主要由主题中的固定层级样式提供（根/分支/叶子）。该方式在浅层结构可读性尚可，但在多层导图中，层级深度与颜色关系不连续，用户难以快速感知深度变化。

本次变更需要在不破坏现有分层边界的前提下，引入“根节点基准色 + 深度色轮轮动”算法：
- 主题系统继续提供根节点基准色与颜色参数。
- 渲染层按节点深度计算默认色。
- 节点显式样式优先级保持不变。

## Goals / Non-Goals

**Goals:**
- 建立统一、可预测的层级背景色算法：以根节点基准色为起点，按固定色相步长逐层轮动。
- 保持同层同色、相邻层级相邻色相，提升层级辨识度。
- 不改变现有数据模型中显式 `backgroundColor` 的覆盖语义。
- 与主题切换、节点重排、文件保存/重开链路保持一致。

**Non-Goals:**
- 不新增用户可配置的色轮步长 UI。
- 不改动节点文字颜色策略与对比度策略（沿用现有主题文本对比机制）。
- 不改变导图文件格式版本或引入迁移脚本。

## Decisions

### 决策 1：采用 HSL 色相轮动，固定步长按深度递进
- 方案：将根节点默认背景色转换到 HSL，保留饱和度/亮度基线，仅对 `Hue` 做 `depth * step` 偏移（取模 `360`）。
- 原因：色相轮动最直观表达“相邻层级相邻颜色”，且计算成本低、结果稳定。
- 备选方案：
  - 预定义离散调色板：实现简单，但当层级较深时易重复且无法保证相邻关系连续。
  - 同色系明暗递进：可读性较高，但“邻层相邻色”语义不明显，不满足需求。

### 决策 2：算法实现放在 shared/theme 侧，渲染层仅消费结果
- 方案：在 `electron/shared/` 或 `src/themes` 对应共享主题工具中提供纯函数 `getNodeDepthBackgroundColor(...)`，渲染层调用该函数而不重复计算逻辑。
- 原因：避免 main/renderer 两侧重复定义领域规则，符合仓库跨进程共享约束。
- 备选方案：
  - 在 Canvas 组件内联计算：改动快，但会造成规则散落、测试困难。

### 决策 3：显式样式优先级不变，自动色只作用于“无显式背景色”节点
- 方案：保留现有 `node.style.backgroundColor` 覆盖默认值的优先级；仅当字段为空时应用层级色轮。
- 原因：避免破坏已存在文档的手动配色意图，降低行为回归风险。
- 备选方案：
  - 强制统一自动色：视觉一致但会破坏用户已有编辑结果，不可接受。

### 决策 4：最小验证覆盖主题切换、层级变化与持久化回放
- 方案：新增/更新单测与集成测试，覆盖以下链路：
  - 主题切换后同一深度颜色可预测
  - 节点深度变化触发颜色重算
  - 保存并重开后颜色表现保持一致
- 原因：这三类场景覆盖大多数真实使用回归风险。

## Risks / Trade-offs

- [风险] 不同主题下根节点亮度差异可能导致部分深度颜色可读性波动  
  → 缓解：沿用现有文本对比度校验与回退策略，不在本次改动中引入第二套文本算法。

- [风险] 深层节点在长链导图中可能出现色相循环重复  
  → 缓解：固定步长并记录循环特性为已知行为；后续如需要可再引入可配置步长能力。

- [风险] 旧测试依赖“固定三层颜色映射”假设导致失败  
  → 缓解：同步更新相关断言为“深度算法预期值”，并保留显式样式覆盖测试。

## Migration Plan

1. 在共享主题工具中新增深度背景色计算函数与常量步长。
2. 替换渲染层节点默认背景色来源为深度计算结果。
3. 运行针对主题、画布、导出与文件重开的最小必要测试集。
4. 若发现回归，可快速回滚到旧默认映射函数，不影响数据文件兼容性。
