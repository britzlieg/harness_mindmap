## Context

本次变更跨越导出链路（renderer -> IPC -> main service）与主题/UI 壳层（renderer 样式 + Electron 窗体），属于多模块联动修复。  
当前主要问题有四类：PNG 导出文本模糊、侧栏编辑输入背景未主题化且缺少玻璃态、静态 `1:1` 文本占用界面、窗体标题栏仍为系统默认样式且未跟随主题。  
根据项目设计文档，依赖方向需保持 `components/hooks -> stores/utils -> preload -> ipc -> services`，避免跨层耦合；同时需满足“画布一致性 + 导出一致性”双检查。

## Goals / Non-Goals

**Goals:**
- 让 PNG 导出在不同缩放倍率下具备可感知的清晰度提升，避免“倍率变化无效”。
- 让侧栏节点编辑输入背景与窗体标题栏控件随主题同步，并维持统一玻璃态视觉语言。
- 清理静态缩放文本展示，降低界面噪音，保持交互能力不变。
- 保持现有文件读写、节点编辑、快捷键与布局行为不回归。

**Non-Goals:**
- 不重构导出格式体系（仅修复 PNG 质量与渲染参数链路）。
- 不新增主题类型或大规模视觉重设计（仅补齐现有主题映射覆盖面）。
- 不引入新的持久化字段或变更 `.mindmap` 文件格式。

## Decisions

### 1) PNG 导出改为“高 DPI 渲染 + 明确像素比传递”
- 决策：在导出服务中统一按 `exportScale * deviceIndependentPixelRatio` 计算实际渲染像素尺寸，并确保文本绘制路径使用同一倍率上下文。
- 原因：当前“比例改变但文字仍糊”通常来自尺寸放大与文本渲染上下文脱节，或快照阶段被二次缩放。
- 备选方案：
  - 仅放大最终输出位图尺寸：实现简单，但文本可能先低分辨率渲染后被拉伸，无法根治。
  - 改为 SVG 再位图化：清晰度好，但会改变现有 PNG 管线与渲染一致性边界，风险更高。

### 2) 侧栏编辑输入改为主题驱动的半透明玻璃态 token
- 决策：新增/复用主题语义 token（输入背景、边框、焦点环、文字），由 `App` 注入 CSS 变量，侧栏编辑输入仅消费变量，不写死颜色。
- 原因：可保证主题切换时即刻同步，避免局部组件维护多套色值。
- 备选方案：
  - 在组件内按主题名硬编码样式：改动集中但可维护性差，易与现有 token 体系冲突。

### 3) 移除静态 `1:1` 缩放文本展示
- 决策：删除 Canvas 控件中的静态比例文案节点，仅保留必要缩放按钮/交互。
- 原因：该文案不承载动态信息且占用注意力，删除可提升界面整洁度。
- 备选方案：
  - 保留但隐藏在窄屏：不能解决桌面端视觉噪音，收益有限。

### 4) Electron 窗体改为主题化自绘标题栏
- 决策：在主进程启用无原生标题栏窗口配置，并在渲染层提供主题化标题栏与最小化/最大化/关闭控件；控件颜色与悬浮态取自主题 token。
- 原因：系统默认标题栏无法跟随应用主题，破坏整体一致性。
- 备选方案：
  - 继续使用系统标题栏，仅调整周边容器：实现成本低，但无法满足“按钮跟随主题”目标。

## Risks / Trade-offs

- [导出性能开销上升] 高倍率与高 DPI 可能增加渲染耗时和内存占用 → 通过上限校验与导出前参数约束控制峰值。
- [跨平台窗体行为差异] 自绘标题栏在 Windows/macOS/Linux 行为可能不同 → 先保证 Windows 路径，保留平台分支与回退策略。
- [样式覆盖冲突] 新增 token 可能与既有玻璃态样式冲突 → 优先扩展现有变量命名体系并补充样式回归测试。
- [误删信息引发可用性争议] 删除 `1:1` 文本后部分用户可能缺少比例提示 → 通过缩放按钮状态和快捷键行为保持可理解性，不改原交互。

## Migration Plan

1. 更新导出参数链路（ExportDialog -> IPC -> export-service）并在服务层完成倍率一致性修复。
2. 扩展主题 token 与侧栏输入样式，验证主题切换即时生效。
3. 替换窗口配置与自绘标题栏控件，接入现有主题状态。
4. 删除静态缩放文本节点并执行 UI 回归。
5. 运行导出、主题、集成相关测试并进行手动视觉验收。

回滚策略：若自绘标题栏引发严重兼容问题，可先回滚到系统标题栏并保留其余三项修复，分阶段上线。
