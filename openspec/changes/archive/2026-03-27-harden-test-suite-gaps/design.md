## Context

当前仓库的测试基线具备一定规模，但信号结构并不均衡：纯算法与导出服务测试较强，跨进程边界、运行时契约和定时/交互类逻辑明显偏弱。现状主要有四个问题：

1. 没有覆盖率统计能力，无法稳定得到覆盖率数字，也无法建立后续门槛。
2. 若干关键边界缺少“直接测试”，例如 `preload`、`assertTrustedIpcSender`、`normalizeWritePath`、`assertMindmapPayload`、`useAutoSave`、`useCanvasInteraction`。
3. 部分集成测试通过大面积 mock 绕开真实边界，导致测试通过并不等于契约真实成立。
4. 失败路径测试会输出大量预期性日志，降低 CI 噪声比。

这次变更横跨 `package.json`、Vitest 配置、renderer 测试、IPC 测试和共享边界模块，属于典型的跨模块测试治理变更，适合先明确设计再实施。

## Goals / Non-Goals

**Goals:**
- 为项目建立可重复执行的覆盖率统计能力，并生成可消费的覆盖率结果。
- 为关键高风险边界补齐直接测试，优先覆盖 `preload`、IPC 安全/路径/载荷校验、自动保存、画布交互。
- 调整脆弱测试结构，减少“被测对象被 mock 掉”的情况，让测试更接近真实契约。
- 清理测试日志噪音，使失败输出更聚焦真实异常。
- 保持 explore 结论与实现任务一一对应，便于后续按 change 落地。

**Non-Goals:**
- 不在本次变更中重构业务实现本身，除非为测试可达性必须做极小范围暴露调整。
- 不在本次变更中引入端到端测试框架或完整 CI 平台改造。
- 不追求一次性把所有文件都补到高覆盖率，而是优先修复关键盲区和信号缺陷。
- 不改变用户可见功能和现有产品行为。

## Decisions

### 决策一：先补“覆盖率能力”，再补“覆盖率数字”

先为 Vitest 增加覆盖率依赖、脚本和输出约定，再基于新能力衡量当前真实基线，而不是先口头设定覆盖率目标。

选择原因：
- 当前缺的不是“目标数字”，而是“可靠测量工具”。
- 没有测量能力时，任何覆盖率承诺都不可验证。

备选方案：
- 直接在 proposal 中写死覆盖率阈值。
  不采用，因为当前没有基线，贸然设阈值容易让首轮实施被阈值问题阻塞。

### 决策二：关键边界优先采用“直接测试”，避免只靠上层间接覆盖

对于 `preload`、IPC 校验函数、路径策略、自动保存、画布交互等边界逻辑，优先补针对模块本身的测试，而不是继续依赖 App 集成测试间接穿透。

选择原因：
- 这些模块承担安全、状态一致性或运行时桥接职责，间接覆盖难以定位问题。
- 上层集成测试常因 mock 过深而绕过真实逻辑。

备选方案：
- 继续增强现有 App 集成测试。
  不采用为主策略，因为它更适合验证用户流，不适合验证安全边界和低层契约。

### 决策三：按“纯函数 / handler / renderer 集成”分层补测

测试补强将按三层推进：
- 纯函数与校验层：`security`、`path-policy`、`validators`
- IPC/bridge 层：`preload`、`layout/file/export` handlers
- renderer 行为层：`useAutoSave`、`useCanvasInteraction`、相关组件集成

选择原因：
- 这和现有代码分层一致，便于定位失败原因。
- 可以减少重复测试同一逻辑，同时提升维护性。

备选方案：
- 统一通过组件集成测试覆盖。
  不采用，因为低层失败会被高层噪声掩盖，且测试会变慢、变脆。

### 决策四：对失败路径日志进行显式控制

对预期失败路径的 `console.error` / `console.log`，在测试中显式捕获、断言或静默处理，避免让正常通过的测试输出大量错误日志。

选择原因：
- 当前测试通过时仍伴随较多日志，降低可读性。
- 失败路径日志本身也是行为的一部分，应该被测试显式管理。

备选方案：
- 保持现状，仅接受噪音。
  不采用，因为 CI 中真正的问题会更难被识别。

## Risks / Trade-offs

- [覆盖率接入后暴露大面积低覆盖模块] → 首轮仅建立能力与基线，不强制一次性全面达标。
- [补测需要少量重构测试可达性] → 仅允许最小范围调整，并优先通过测试层解决。
- [减少 mock 后测试运行变慢] → 优先在关键边界使用直接测试，避免把所有测试都提升为重集成测试。
- [日志静默可能掩盖真实问题] → 仅对预期失败路径做定向控制，并保留断言。

## Migration Plan

1. 为 Vitest 补齐 coverage 依赖、脚本与报告输出约定。
2. 新增关键边界测试文件，优先覆盖 security/path-policy/validators/preload/layout handler。
3. 为 `useAutoSave` 与 `useCanvasInteraction` 增加更贴近真实行为的测试。
4. 收敛现有集成测试中的深度 mock，保留必要的上层回归验证。
5. 清理测试日志噪音，并以一次全量测试加一次 coverage 运行验证结果。

回滚策略：
- 若覆盖率接入或新测试结构导致短期不稳定，可先保留测试文件，回退覆盖率门槛要求，而不回退测试本身。

## Open Questions

- 首轮是否仅输出 coverage 报告，还是同步引入最低阈值校验。
- `preload` 测试是以模块级 mock `electron` 为主，还是增加更强的契约快照校验。
- `useCanvasInteraction` 是否需要补到事件细节级别，还是以 store 结果断言为主。
