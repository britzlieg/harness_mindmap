## 上下文

当前应用已支持三种布局模式（`mindmap`、`logic`、`org`），其算法分发位于 `src/utils/layout-algorithms.ts`，类型约束位于 `src/types/index.ts`，选择入口在 `src/components/Sidebar/LayoutSelector.tsx`。现有模式无法满足部分用户的单向树状浏览和层级梳理需求，且当文件元数据出现不支持的布局值时，布局意图可能丢失。

本次变更在保持文件兼容性的前提下引入更多布局模式，并保证布局计算结果稳定可预测，便于测试与维护。

## 目标 / 非目标

**目标：**
- 至少新增两种布局模式，并提供确定性的节点排布规则。
- 保持侧边栏布局选择入口可发现，并持续提供明确的激活态反馈。
- 将用户选择的布局模式持久化到元数据，并在加载时恢复。
- 对无效/未知布局值提供安全回退策略。
- 为新算法与布局管理行为补充自动化测试。

**非目标：**
- 不实现自由拖拽式的手工自定义布局编辑。
- 不进行与布局选择无关的主题系统或视觉重构。
- 不改变现有 `mindmap`、`logic`、`org` 的核心语义。

## 关键决策

### 决策 1：新增 `tree-right` 与 `tree-left` 两种内置布局

理由：
- 能覆盖常见的单向结构整理场景，用户无需通过旋转或替代布局来迁就阅读方向。
- 可复用现有子树测量思路，降低实现风险与维护成本。

备选方案：
- 只新增一种单向树布局：不采纳，因为仍无法覆盖镜像阅读方向需求。
- 优先做径向布局：暂不采纳，因为会引入更大范围的连线与渲染调整。

### 决策 2：在统一分发路径中处理布局分发与回退

理由：
- `computeLayout` 应保持为布局算法选择的单一入口。
- 未知布局值统一回退到 `mindmap`，保证不同版本文件均可打开。

备选方案：
- 仅在 UI 层校验：不采纳，因为文件导入与 IPC 路径可绕过 UI。

### 决策 3：将布局模式视为可持久化的文档核心数据

理由：
- `layoutType` 属于文档语义的一部分，必须贯穿 create/open/save/export 全流程。
- 新模式接入需要同时更新类型定义、选择器选项与元数据合法性处理逻辑。

备选方案：
- 保持松散字符串类型：不采纳，因为会降低类型安全并提升运行时错误概率。

## 风险 / 权衡

- [风险] 深层树结构下可能出现节点重叠。
  -> 缓解：定义最小间距约束，并为深树/宽树场景补充算法单元测试。

- [风险] 存量文件可能包含未知 `layoutType`。
  -> 缓解：在加载与分发路径中统一回退到 `mindmap`，并增加对应测试。

- [风险] 布局选项增加后，UI 可能显得拥挤。
  -> 缓解：保持简洁命名并按高频使用顺序展示。

## 迁移计划

1. 扩展 `LayoutType` 与布局选择器选项，加入新模式。
2. 在 `layout-algorithms.ts` 中实现并接入新算法分发。
3. 在加载路径补充元数据合法性守卫与回退逻辑。
4. 补充测试，覆盖：
   - 新布局算法输出；
   - 新模式与未知模式的分发行为；
   - UI 选择与元数据更新行为。
5. 发布时无需数据迁移；若回滚，未知模式将安全回退到 `mindmap`。
