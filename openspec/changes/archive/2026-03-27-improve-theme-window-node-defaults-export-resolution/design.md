## Context

当前思维导图应用已支持主题切换、文件保存与 PNG 导出，但仍存在三类体验断层：

1. 窗体层（应用壳）未接入主题系统，视觉上与画布不一致。
2. 节点文本允许为空，导致保存与导出后出现无语义节点。
3. PNG 导出分辨率固定，复杂导图文字模糊且不可调。

该变更跨越主题、节点数据、导出流程三个模块，属于多模块协同改动，需要在实现前统一约束与数据流。

## Goals / Non-Goals

**Goals:**

- 让程序窗体样式与当前导图主题保持一致，并在主题切换时即时生效。
- 建立节点默认文本与导出/保存前兜底策略，避免任何空文本节点落盘或进入导出图像。
- 在 PNG 导出前提供缩放百分比输入，控制导出清晰度，并保证参数合法性与可预期结果。

**Non-Goals:**

- 不重构现有主题定义结构（仅复用与扩展已有主题变量）。
- 不新增除 PNG 之外的导出格式。
- 不引入复杂排版增强（如抗锯齿参数面板、多档预设模板等）。

## Decisions

### 决策 1：窗体主题采用“主题变量映射 + 根容器 class”方案

- 方案：将当前主题映射到应用根容器（或窗体包装层）class 与 CSS 变量，窗体背景、边框、阴影、标题区域使用同一主题 token。
- 原因：与现有画布主题机制一致，避免重复维护两套颜色系统。
- 备选方案：
  - 直接在组件内硬编码颜色：实现快，但无法统一维护，后续扩展成本高。
  - 运行时注入独立 style 标签：灵活但复杂度高，不利于测试。

### 决策 2：节点默认文本采用“双阶段兜底”

- 方案：在“新建节点”时注入默认文本（如“新节点”）；在“保存/导出前序列化”再执行一次空文本修复，确保历史数据或外部导入节点也被兜底。
- 原因：仅在创建时兜底无法覆盖旧数据；仅在保存时兜底会导致编辑过程出现空节点体验。
- 备选方案：
  - 只在 UI 输入层禁止空值：无法覆盖程序化创建和历史数据。
  - 仅在落盘时替换：用户编辑时仍可见空节点，交互不稳定。

### 决策 3：PNG 导出缩放值使用“百分比输入 -> 导出倍率”

- 方案：导出弹窗新增缩放百分比输入框，默认 100；允许范围 50-400；导出时将百分比转换为倍率（`scale = percent / 100`）传入导出渲染管线。
- 原因：百分比符合用户直觉，且对清晰度调节足够细粒度。
- 备选方案：
  - 下拉固定档位（100/200/300）：实现简单但灵活性不足。
  - 直接输入倍率（1.0/2.0）：技术导向，不利于普通用户理解。

### 决策 4：导出前参数校验前置在弹窗确认阶段

- 方案：用户点击确认时先做整数校验与范围校验，不通过则阻断导出并提示；通过后再调用导出 IPC。
- 原因：尽早失败，减少主进程无效任务；错误反馈更直接。
- 备选方案：
  - 在主进程二次校验后报错：安全但反馈链路更长，交互延迟更明显。

## Risks / Trade-offs

- [风险] 主题 token 不完整导致窗体局部仍显示默认样式。  
  -> 缓解：梳理窗体层必要 token，缺失项提供主题级默认值并补测试快照。

- [风险] 默认文本策略可能覆盖用户有意设置的空白内容。  
  -> 缓解：将“空文本”定义为 `null/undefined/trim 后为空`，仅对该集合兜底，不改动有效文本。

- [风险] 高缩放（如 400%）在大图下导出耗时增加。  
  -> 缓解：设置上限并在导出中展示进行中状态；保留后续分块导出的扩展点。

## Migration Plan

1. 先上线节点默认文本兜底与导出参数校验（低风险、可快速验证）。
2. 再接入窗体主题联动，补充视觉回归测试。
3. 发布后通过典型导图样本验证 100%、200%、300%、400% 导出清晰度与性能。
4. 回滚策略：保留旧导出入口参数分支，必要时可临时固定回 100% 导出并关闭自定义输入。

## Open Questions
