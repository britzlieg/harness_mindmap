## Context

现有主题系统已经覆盖画布与部分壳层样式，但菜单与侧栏文字色在不同主题下仍沿用通用色值，导致在高饱和背景或高亮主题中出现刺眼与对比不足。  
本次改动涉及主题 token、全局样式变量、菜单/侧栏组件，属于跨模块视觉规范调整，需要在实现前统一颜色映射与回归标准。

## Goals / Non-Goals

**Goals:**

- 基于现有 UI 色轮思路，为菜单与侧栏建立新的字体色映射规则，确保在所有内置主题下可读。
- 将文字色分层为主文本、次文本、禁用文本、强调文本，避免单一色值覆盖所有语义层级。
- 建立可回归的可读性约束（对比度下限与视觉舒适度）并纳入测试。

**Non-Goals:**

- 不新增主题数量或重做整套主题系统结构。
- 不调整节点内容区（画布节点文本）的主题配色策略。
- 不引入新的第三方色彩库或设计系统依赖。

## Decisions

### 决策 1：新增“文字语义 token”而不是直接在组件硬编码颜色

- 方案：在主题变量层新增/统一 `--ui-color-text`、`--ui-color-text-muted`、`--ui-color-text-soft`、`--ui-color-accent-text` 等语义 token，菜单与侧栏组件只消费语义 token。
- 原因：可统一回归，后续主题扩展时只改 token 映射，不改组件逻辑。
- 备选：
  - 组件内逐个改色：短期快，但维护成本高且容易不一致。

### 决策 2：采用“主题主色 + 色轮补偿”生成字体色

- 方案：以当前主题主色为基准，结合背景亮度做色轮补偿（邻近/对比色）得到字体色层级，优先保证可读性，再考虑风格一致性。
- 原因：能同时满足“参考色轮设计”与“可读性稳定”。
- 备选：
  - 固定黑白两套文字：对部分主题不刺眼，但风格割裂且层级不足。

### 决策 3：加入对比度阈值回退机制

- 方案：若计算得到的字体色与背景色对比度低于阈值，则自动回退到安全文字色集合（浅色背景回退深灰，深色背景回退浅灰/白）。
- 原因：避免极端主题下出现不可读文本。
- 备选：
  - 只依赖人工挑色：上线后仍可能出现遗漏。

## Risks / Trade-offs

- [风险] 色轮补偿算法在个别主题下仍可能偏艳。  
  -> 缓解：限制饱和度与亮度范围，并提供安全回退色。

- [风险] 全局 token 调整可能影响非目标组件。  
  -> 缓解：先限定菜单与侧栏使用的新 token，逐步推广，配套回归测试。

- [风险] 可读性标准过严会削弱主题风格差异。  
  -> 缓解：在满足阈值前提下保留主题主色倾向，仅在不达标时回退。

## Migration Plan

1. 先补充主题文字 token 和色轮映射逻辑。  
2. 将菜单与侧栏组件切换到新 token。  
3. 执行主题回归测试并人工检查高风险主题（深色/高饱和主题）。  
4. 如出现视觉争议，可快速回退为“安全文字色集合”配置。

## Open Questions

- 对比度阈值是否统一采用 WCAG AA 4.5:1，还是对小字与大字分级？  
- 是否需要在主题编辑器中暴露“文字强度”调节项（后续需求）？

## Implementation Notes

- 已采用语义字体色 token：`text-primary`、`text-secondary`、`text-disabled`、`text-accent`。
- 主题映射采用“主色互补 + 邻近色”色轮策略，并对高饱和主题执行饱和度压缩，降低刺眼感。
- 可读性校验使用对比度阈值：主文本与强调文本 `>= 4.5`，次文本 `>= 3.3`，禁用文本 `>= 2.6`。
- 当映射结果不达标时自动回退到安全明暗文字色，确保菜单与侧栏可读性稳定。
