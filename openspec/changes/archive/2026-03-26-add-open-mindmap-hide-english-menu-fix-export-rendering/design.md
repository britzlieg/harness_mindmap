## 背景

桌面应用已支持新建/保存思维导图、中文界面标签以及 PNG 导出，但仍存在三个用户可感知问题：
- 用户无法通过明确的“打开”操作稳定地重新加载已保存的 `.mindmap` 文件；
- Electron 原生顶层菜单仍暴露英文标签（`File`、`Edit` 等）；
- PNG 导出结果可能缺少连线和文字，出现内容不完整。

这些问题同时涉及渲染进程与主进程（菜单初始化、IPC 文件处理、导出链路），在实现前需要统一技术方案。

## 目标 / 非目标

**目标：**
- 提供一等公民的“打开”命令，用于选择并加载 `.mindmap` 文件到当前编辑会话。
- 在最终产品体验中隐藏英文顶层原生应用菜单。
- 确保 PNG 导出包含完整可视思维导图内容，包括节点文字与连线。
- 在打开/导出失败时保持应用稳定并保护当前会话状态。

**非目标：**
- 不重构除“打开”相关入口以外的整体工具栏/菜单信息架构。
- 不修改 `.mindmap` 文件格式。
- 不新增除 PNG 之外的其他导出格式。

## 关键决策

1. 通过共享 IPC 协议支持打开文件
- 决策：新增或扩展主进程处理器，触发 `.mindmap` 原生文件选择，并将校验后的序列化数据返回给渲染进程状态加载器。
- 原因：保持文件系统高权限访问在 Electron 主进程内，同时复用现有保存/加载的数据解析模式。
- 备选方案：仅在渲染进程通过浏览器 API 读取文件。否决原因是与现有桌面权限模型和 IPC 架构不一致。

2. 在窗口级隐藏英文原生菜单
- 决策：在桌面启动阶段禁用或清空应用菜单（`Menu.setApplicationMenu(null)` 或等效平台安全方案），同时保留应用内中文命令入口。
- 原因：可在不同平台下稳定保证不显示英文 `File/Edit/...` 顶层菜单。
- 备选方案：将原生菜单改为中文。否决原因是本次需求明确要求“隐藏英文菜单”，不是“翻译菜单”。

3. 从完整渲染层导出 PNG 快照
- 决策：PNG 导出改为从与用户可见一致的最终渲染层截取，并在快照前等待渲染完成。
- 原因：当前“缺少连线/文字”现象表明导出可能读取了不完整图层，或与渲染时序存在竞态。
- 备选方案：基于原始数据重建一套独立绘制导出逻辑。否决原因是重复实现成本高，且易与屏幕显示产生偏差。

4. 打开与导出错误采用会话保护策略
- 决策：统一错误处理，打开/导出失败返回结构化错误、保留当前内存中的思维导图，并提供用户可见反馈。
- 原因：避免崩溃及部分状态被破坏的风险。

## 风险 / 权衡

- [风险] 不同平台（macOS / Windows / Linux）菜单行为存在差异，可能影响隐藏策略一致性。
  [缓解] 增加平台相关菜单初始化测试与运行时断言，确认无英文顶层菜单可见。

- [风险] 增加“等待渲染就绪”可能带来导出延迟感知。
  [缓解] 采用有界等待策略，仅等待必要渲染完成信号，并保留导出中的用户反馈。

- [风险] 老版本或损坏的 `.mindmap` 文件在打开时可能解析失败。
  [缓解] 增加版本/结构校验，失败时优雅降级并保持当前编辑内容不变。

## 迁移计划

- 无需数据迁移。
- 以代码方式发布：主进程、渲染进程与测试一并落地。
- 回滚策略：回退本次变更集，可恢复现有新建/保存/导出基线行为。
