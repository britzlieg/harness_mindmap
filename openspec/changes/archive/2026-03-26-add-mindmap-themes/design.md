## Context

当前代码已经具备主题资源与主题选择 UI（`src/themes/index.ts`、`ThemeSelector.tsx`），并在元数据中保存 `theme` 字段，但主题未贯穿画布渲染路径：
- `MindMapCanvas` 仍使用固定背景色与连线样式。
- `NodeRenderer` 主要依赖 `node.style`，未按节点层级套用主题默认样式。
- 主题切换与渲染之间缺少统一的“已解析主题对象”输入。

该变更需要在不破坏现有节点自定义样式的前提下，让主题成为可预期、可持久化的视觉配置。

## Goals / Non-Goals

**Goals:**
- 让已选主题驱动画布背景、网格、连线和节点默认样式。
- 定义稳定的样式优先级：`node.style` 显式值 > 主题层级默认值 > 组件硬编码兜底。
- 保持主题保存/重开一致性，并对未知主题值安全回退到 `default`。
- 为关键行为补齐测试，覆盖切换、回退与渲染结果。

**Non-Goals:**
- 不在本次引入用户自定义主题编辑器。
- 不改动文件格式版本与数据库 schema（沿用现有 `metadata.theme`）。
- 不改变布局算法、节点数据结构或导出文件协议。

## Decisions

### 1) 在画布层统一解析主题并向下传递
- 决策：在 `MindMapCanvas` 基于 `metadata.theme` 调用 `getTheme` 得到 `activeTheme`，并将其拆分给 `ConnectionCanvas` 与 `NodeRenderer`。
- 原因：避免各组件重复回退逻辑，降低主题切换时的渲染分叉风险。
- 备选方案：在每个组件内单独读取 store 并解析主题。未选原因：耦合更高、难以统一测试。

### 2) 节点样式采用“分字段合并”而非整对象覆盖
- 决策：按节点层级（root/branch/leaf）先取主题样式，再按字段覆盖 `node.style` 中已定义值。
- 原因：保障用户手动设置不会被主题切换抹掉，同时允许主题补齐缺失字段。
- 备选方案：切换主题时直接写回所有 `node.style`。未选原因：不可逆、污染用户数据、撤销/重做复杂。

### 3) 未知主题值在运行时回退并保持可保存
- 决策：渲染与读取路径均通过 `getTheme` 回退到 `default`；UI 高亮也基于回退后的主题 key。
- 原因：旧文件或外部编辑带来的非法主题值不应阻塞打开与编辑。
- 备选方案：加载时强制改写元数据主题值。未选原因：会引入隐式数据变更，增加保存副作用。

### 4) 最小化改动连接与画布组件接口
- 决策：`ConnectionCanvas` 继续接收基础样式 props，但其值由 `MindMapCanvas` 主题化提供；画布背景与网格颜色同理。
- 原因：降低组件重构范围，减少回归风险。
- 备选方案：引入全局 Theme Context。未选原因：当前规模下收益有限，增加额外抽象成本。

## Risks / Trade-offs

- [Risk] 主题切换频繁触发重渲染，影响大图性能
  - Mitigation: 保持主题对象引用稳定（按 key 解析），并尽量复用现有渲染缓存路径。
- [Risk] 样式合并规则实现不一致，导致节点视觉漂移
  - Mitigation: 抽离共享样式合并函数并做单元测试。
- [Risk] 既有测试基于硬编码颜色，改动后失败
  - Mitigation: 更新断言为主题驱动值，补充针对 `default` 回退场景的测试。

## Migration Plan

1. 先落地规格与测试用例，锁定期望行为。
2. 实现渲染链路主题注入与样式合并。
3. 运行前端与数据层相关测试，确保保存/重开不回归。
4. 若出现严重视觉回归，可临时回退到固定渲染参数（保留 `metadata.theme` 存储不变）。

## Open Questions

- 主题是否需要影响导出（PNG/SVG）视觉风格？本变更暂不覆盖，后续可在导出能力中补充。
- 主题选择器是否需要按主题分组或搜索？当前规模不需要，后续随主题数量增长再评估。
