## 背景

当前桌面应用已经具备部分文件与导出基础能力：
- 文件 IPC 已支持 `create`、`open`、`save`、`saveAs`。
- 渲染层已注册 `Ctrl/Cmd+N/O/S`，但工具栏动作与保存逻辑尚未完全统一。
- 导出弹窗当前支持 Markdown 与 SVG；IPC 过滤器虽声明了 PNG，但端到端导出尚未落地。
- 用户可见的核心操作文案在工具栏与导出相关界面中仍以英文为主。

本次变更需要在不引入完整 i18n 框架的前提下补齐核心操作流程。

## 目标 / 非目标

**目标：**
- 提供一致的新建/保存流程，确保工具栏与快捷键触发行为统一。
- 在现有导出流程中提供可实际使用的 PNG 导出能力。
- 将工具栏、导出弹窗和相关控制区的核心菜单/动作文案改为中文。
- 保持现有文件格式与快捷键行为不变。

**非目标：**
- 不实现完整的多语言运行时切换能力。
- 不改造脑图底层数据模型或 SQLite schema。
- 不对导出布局/视觉风格做超出当前基线的重设计。

## 关键决策

### 决策 1：复用现有文件 IPC 并统一渲染层入口
- 复用现有 `window.electronAPI.file.create/save` 接口契约。
- 在工具栏补齐或接入显式保存动作，并复用与快捷键一致的处理逻辑。
- 保持 `Ctrl/Cmd+N/O/S` 绑定不变，仅统一命令分发入口。

备选方案：
- 方案：围绕集中式命令总线重做文件命令处理。
- 结论：本次不采用，因其显著扩大范围，且不能优先解决当前流程断点。

### 决策 2：在 Electron 导出服务中实现 PNG 导出
- 在 Electron 导出服务中扩展 PNG 生成功能，基于当前节点树生成稳定输出。
- 继续以 `export:saveAs` 作为统一选路入口，在 `format === 'png'` 时进入 PNG 分支。
- 保持 Markdown/SVG 现有行为不变。

备选方案：
- 方案：在渲染层通过截图/画布捕获生成 PNG。
- 结论：本次不采用，因为导出职责应位于 Electron 主进程，且渲染层捕获会耦合视口与 UI 状态。

### 决策 3：通过局部 UI 文案常量完成中文化
- 在相关 UI 组件中用中文替换硬编码英文菜单/动作文案。
- 为保持代码稳定性，内部标识符和 API 字段名仍保留英文。
- 采用局部文案常量/映射方案，不引入 i18n 依赖。

备选方案：
- 方案：引入完整 i18n 依赖与 locale 文件体系。
- 结论：本次不采用，因为当前需求仅涉及中文化，不需要完整多语言架构。

## 风险 / 权衡

- [风险] PNG 导出结果与屏幕显示存在视觉差异。 -> [缓解] 固定导出画布尺寸并通过样例测试校验。
- [风险] 中文化改造可能遗漏分散的硬编码字符串。 -> [缓解] 在受影响组件中做文案盘点，并增加 UI 文案断言测试。
- [风险] 统一保存入口后，可能与自动保存/快捷键出现重复触发。 -> [缓解] 保证保存操作幂等，并验证快捷键与工具栏触发路径。

## 迁移计划

1. 实现渲染层命令接线，统一新建/保存触发路径。
2. 在 Electron 导出处理与服务层补齐 PNG 导出分支。
3. 将目标菜单/动作文案替换为中文。
4. 增加或调整测试，覆盖文件流程、PNG 选项与中文文案。
5. 发布时无需数据迁移；若回滚，直接回退 UI 与处理器改动。
