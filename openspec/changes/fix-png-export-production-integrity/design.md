## Context

当前 PNG 导出在代码结构上已经形成了两条并存路径：生产导出请求主要经过 `export-handlers -> export-orchestrator -> scene-builder -> png-renderer`，而现有较多测试仍直接覆盖 `export-service` 兼容层。根据 `docs/mindmap-app-architecture/06-png-export-integrity-fix.md` 的排查结论，当前最可信的问题不是单点缺陷，而是多个链路差异叠加后的结构性偏差：

- `fitSceneForRaster` 在已识别内容越过右/下边界时，仍可能只按 `right - left` 与 `bottom - top` 计算扩容，导致画布并未真正覆盖绝对边界。
- Electron 分块捕获对尺寸不足的边缘瓦片会直接 `resize` 后再拼接，这会用插值放大局部瓦片，制造局部模糊。
- Electron 光栅化路径未稳定通过校验时，系统会频繁降级到 fallback，而 fallback 的伪字模渲染天然难以达到真实字体渲染的清晰度。
- 生产链路与兼容链路对 `scale` 的处理并不完全一致，导致“规范、实现、测试”之间出现偏移。

这次变更跨越导出服务、兼容层、测试与日志观测，属于典型的跨模块修复；同时它直接影响 Electron 主进程导出行为，需要遵守仓库现有分层约束，不在 renderer 与 main 两侧重复定义领域协议。

## Goals / Non-Goals

**Goals:**

- 修复 PNG 栅格场景的边界扩容逻辑，确保正坐标轻微溢出右侧或底部时不会再发生截断。
- 收紧分块捕获的瓦片校验，禁止通过静默拉伸过小瓦片来“伪造完整图”，优先保证结果可信。
- 让 PNG 导出链路的路径命中与 fallback 原因可观测，便于测试、排查和后续回归。
- 对齐生产链路与兼容层的倍率处理语义，确保不同入口的导出行为一致。
- 将测试重心补到 `export-orchestrator -> png-renderer` 真实生产链路，同时保留兼容层的最小回归保护。

**Non-Goals:**

- 不在本次变更中重写软件 fallback 的文字引擎，也不承诺让 fallback 达到原生字体渲染同等级清晰度。
- 不修改导出弹窗交互、导出格式集合或非 PNG 导出链路。
- 不引入新的外部渲染依赖或替换 Electron 现有光栅化方案。

## Decisions

### 决策一：边界扩容改为基于绝对栅格坐标归一化

在 `fitSceneForRaster` 中，将“内容边界”与“当前画布边界”统一映射到同一组绝对栅格坐标后再计算扩容结果。新宽高不再仅依赖 `right - left` 或 `bottom - top`，而是必须保证扩容后的画布右边界覆盖 `max(currentRight, contentBounds.right)`，底边界覆盖 `max(currentBottom, contentBounds.bottom)`，同时兼容左/上方向的负坐标偏移。

选择这个方案，是因为它直接修正了“已经检测到越界却没有真正扩画布”的根因；相比继续增加固定 padding 或导出后再裁切，这种做法能从边界模型层面保证正确性。

备选方案：

- 增加默认安全边距。未采用，因为它只能降低复现概率，不能证明不会截断。
- 先渲染再根据像素内容二次裁切。未采用，因为无法弥补第一次渲染时已经发生的内容丢失。

### 决策二：尺寸不足的瓦片视为捕获失败，而不是可拉伸资源

在 `png-renderer` 的 tiled capture 路径中，边缘瓦片如果小于预期尺寸，不再直接 `resize` 到目标尺寸后参与拼接。新的处理方式是：

1. 记录尺寸不足的瓦片信息与当前导出上下文。
2. 视情况执行一次受控重抓，或直接进入明确的 fallback / 失败分支。
3. 仅允许裁剪过大的瓦片，不允许放大过小的瓦片。

选择这个方案，是因为“静默拉伸”会把尺寸问题转化为视觉模糊，掩盖真实错误来源。与其输出一张“完整但糊”的 PNG，不如明确地重试或降级，并把原因记录清楚。

备选方案：

- 保留 `resize`，仅调整插值算法。未采用，因为无论使用何种插值，都会改变局部文字与线条的真实像素密度。
- 遇到尺寸不足直接抛错终止。暂不直接采用，因为受控 fallback 能维持更好的用户兜底体验。

### 决策三：为渲染结果引入结构化诊断信息，但默认不暴露到 renderer UI

本次变更会在 main 进程导出链路内部引入结构化诊断信息，例如：

- 命中的渲染路径：单帧 Electron、分块 Electron、软件 fallback
- fallback 原因：尺寸校验失败、内容校验失败、异常捕获失败、重抓后仍失败
- 分块捕获摘要：瓦片总数、失败瓦片、是否发生重抓
- 归一化后的倍率来源与最终像素尺寸

这些信息优先用于日志和测试断言，不默认进入 renderer UI，也不强制扩展用户可见的 IPC 返回值；如果需要跨进程共享枚举或结构，再放到 `electron/shared/` 中统一定义。

选择这个方案，是为了兼顾可观测性与最小改动边界。直接把诊断细节暴露给前端会扩大协议面，不符合当前“最小且符合分层边界”的原则。

备选方案：

- 只打文本日志，不定义结构。未采用，因为这会让测试断言脆弱，也不利于后续统计。
- 将完整诊断对象回传 renderer。未采用，因为当前用户流程并不需要消费这类细粒度内部状态。

### 决策四：倍率归一化只保留一套语义，并由生产链路复用

将 PNG 导出倍率的归一化与安全收敛逻辑提取为单一来源，确保 `export-orchestrator` 主路径与 `export-service` 兼容层都使用同一套倍率计算与传参语义。`buildScene` 负责构建已应用倍率的场景数据，`renderPng` 接收统一归一化后的导出上下文，避免“某条链路只看场景倍率、另一条链路还额外读取原始 scale”的分叉。

选择这个方案，是因为倍率行为的分叉会直接削弱规范的可验证性。统一入口后，测试才能真正证明 `100%`、`200%`、`300%` 等倍率在不同调用路径下是一致的。

备选方案：

- 保持兼容层单独处理 `scale`。未采用，因为它会继续制造生产链路与测试链路偏差。

### 决策五：测试分层改为“生产链路优先，兼容层兜底”

测试策略调整为：

- 单元测试聚焦 `fitSceneForRaster` 与瓦片校验辅助逻辑。
- 集成测试优先覆盖 `export-orchestrator -> png-renderer` 的真实生产链路，并显式断言命中路径或 fallback 原因。
- 兼容层 `export-service` 保留少量行为回归测试，防止旧调用入口失效。

选择这个方案，是因为当前问题的根源之一就是测试覆盖面偏离真实生产入口。把主要断言迁到生产链路，才能把“绿测”重新和“真实可用”绑定起来。

## Risks / Trade-offs

- [更严格的瓦片校验可能提升 fallback 频率] → 通过受控重抓与结构化诊断降低误判，并在测试中覆盖常见边缘瓦片尺寸。
- [内部诊断字段增加后，日志与测试存在同步维护成本] → 通过共享枚举、固定事件名与最小字段集控制变更面。
- [统一倍率语义后，旧兼容测试可能出现行为变化] → 先补生产链路断言，再调整兼容层测试基线，确保变化是显式且可解释的。
- [Electron 环境测试仍可能受平台差异影响] → 让大部分断言建立在可注入的渲染结果与诊断摘要上，而不是依赖脆弱的像素偶然性。

## Migration Plan

1. 先补 OpenSpec 变更文档，明确边界扩容、瓦片校验、路径观测与倍率一致性的目标行为。
2. 在 `png-renderer` 与相关辅助函数中引入绝对边界归一化、瓦片尺寸校验与诊断摘要生成。
3. 调整 `export-orchestrator` 与 `export-service` 的倍率传递与结果封装，让两条入口共享同一套归一化语义。
4. 补充与迁移测试，优先让 `export-orchestrator -> png-renderer` 成为主验证入口。
5. 以最小测试集验证通过后再实施后续代码修复；如果新逻辑在真实环境中引发过高 fallback，可回退到旧的捕获流程，但保留新增诊断与测试，以便继续定位。

本次变更不涉及用户数据迁移，也不需要修改持久化格式。

## Open Questions

- 边缘瓦片尺寸不足时，受控重抓的次数应固定为一次，还是按瓦片类型做差异化策略，需要在实现时结合当前 Electron 行为再定。
- 结构化诊断信息是否需要在未来通过调试开关透出给前端开发工具，本次先不纳入范围，但实现时应避免把路径封死。
